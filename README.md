# Progressive Overload

Workout tracker that logs sets/reps/weight per movement and recommends the next
session's load and rep targets. **Pounds only. 45 lb bars only.** No metric path,
no `bar_weight_lb` column — those are deliberate non-goals.

This repo implements **all ten steps** of the build spec: the foundations, the
progression engine, auth + persistence, the Log Set + Exercise Detail screens, the
feature-extraction pipeline / training table, the FastAPI ML service, the blend →
safety-rails → rounding recommendation wired into the app, and the feedback loop /
nightly retrain gate. The progression trigger follows [PROGRESSION.md](PROGRESSION.md),
an evidence-based companion spec that **supersedes** the naive Phase 1 rules.

The Log Set target is the **real recommendation**: rule engine (readiness +
vetoes) → optional ML magnitude blend (`α·ml + (1−α)·rule`, α=0 during cold start)
→ safety rails → floor-to-loadable. If the ML service is unset or unreachable the
blend uses α=0 — the rule engine is the permanent fallback and logging never breaks.

**Feedback loop (step 10):** every recommendation is logged with both predictions;
its outcome (accepted + realized e1RM) is recorded when the session finishes. The
nightly [ml/evaluate.py](ml/evaluate.py) job compares ML vs rule MAE on each user's
held-out last 3 sessions and, when the model loses, sets that user's `ml_alpha_cap`
to 0 — which the blend clamps α to. ML has to keep earning its place.

### Offline-first ([OFFLINE_FIRST.md](OFFLINE_FIRST.md))

The recommendation and logging paths never touch the network. [LocalFirstStore](src/data/localStore.ts)
uses **IndexedDB** as the system of record: every read is local, every write lands
locally first with a client-generated UUID and enqueues an idempotent sync op.
[Home](src/screens/Home.tsx) renders the target from pure local TS immediately;
ML is a background refinement only. On finishing a session the *next* session's
prescription is precomputed and persisted, so a cold open reads a ready value.
`flush()` drains the sync queue to Supabase (upsert-by-id, last-write-wins) on
reconnect.

The app is a **PWA** ([vite-plugin-pwa](vite.config.ts)): a workbox service worker
precaches the app shell (HTML/JS/CSS/icon/manifest) and a web manifest makes it
installable with `display: standalone`. Shell-cache + IndexedDB together give a
full offline launch — installed, it opens and logs a workout with zero network.

**Within-session autoregulation** ([liveProgression.ts](src/lib/liveProgression.ts)):
after each logged set the next set's target advances instantly and locally — one
loadable increment up when you beat it easily, one down on a clear miss — plus an
effort-scaled rest timer. Pure local TS; the heavy between-session math still runs
only on finish.

**Returning-user hydration** ([remoteSource.ts](src/data/remoteSource.ts)): on
login the store flushes local pending writes, then pulls the server's truth down
(upsert-by-id, never deleting local-only rows) so a fresh device fills from Supabase
without ever blocking the live path.

### ML service ([ml/](ml/))

A Python/FastAPI service exposing `POST /predict` (next-session e1RM regression +
rep-completion probability) and `GET /health`. Interpretable ElasticNet by default
(gradient-boosting optional), with a heuristic fallback so it always answers. The
feature layout is shared with the TS pipeline through [ml/feature_spec.json](ml/feature_spec.json)
(generated by `npm run feature:spec`; a TS drift guard fails if they diverge).
Blending / safety rails / rounding stay on the TS side — the rule engine is the
permanent fallback. See [ml/README.md](ml/README.md).

### Running modes

- **Demo mode** (default, zero config): no Supabase env → an in-memory store with
  a seeded catalog + squat history, and a local fake user. `npm run dev` just works.
- **Supabase mode**: follow [SUPABASE_SETUP.md](SUPABASE_SETUP.md) — create a
  project, `supabase db push` the migrations, and set `VITE_SUPABASE_URL` +
  `VITE_SUPABASE_ANON_KEY`. The same screens then read/write real rows under RLS.

The UI depends only on the `WorkoutStore` interface ([src/data/store.ts](src/data/store.ts));
`InMemoryWorkoutStore` and `SupabaseWorkoutStore` are interchangeable behind it.

## Commands

```bash
npm install
npm run dev           # Vite dev server (Log Set demo screen)
npm run build         # typecheck + production bundle
npm test              # 91 unit + component tests
npm run typecheck     # tsc --noEmit
npm run seed:validate # validate seeds/exercises.json
npm run seed:sql      # regenerate the seed migration from the JSON
npm run feature:spec  # regenerate ml/feature_spec.json from FEATURE_NAMES
npm run db:check      # validate migrations against the real PostgreSQL grammar
```

ML service (from `ml/`): `pytest`, `python train.py --n 4000`, `uvicorn app.main:app`,
`python evaluate.py --input outcomes.json` (nightly gate; `--retrain table.json` to retrain first).

## Two invariants that govern everything

1. **All weights are pounds.** No conversion layer anywhere.
2. **All barbells are 45 lb.** `equipment = 'barbell'` implies a 45 lb bar; there
   is no override column. EZ / trap bars are out of scope for v1 precisely
   because they'd need one.

## Load math — `effectiveLoad()`

Every tonnage, e1RM, and ML feature routes through
[`effectiveLoad()`](src/lib/effectiveLoad.ts). `load_type` decides how the logged
`weight_lb` maps to load on the body:

- `total` → as logged (barbell, most machines)
- `per_hand` → ×2 (two dumbbells)
- `per_side` → ×2 (plate-loaded machines)
- `bodyweight_plus` → bodyweight + logged (dips, pull-ups)

Mislabeling `per_side` as `total` silently halves tonnage — the validator flags
the obvious equipment/load_type mismatches, but the field still deserves care.

## Rounding & loadability

Hard invariant: **every emitted target is an exact multiple of 2.5 lb.** The
global rounding rule and the barbell constraint are the same constraint. Order of
operations for a recommendation (Phase 2+) is fixed:

```
raw    = blend(ml, rule)
capped = applySafetyRails(raw)
final  = snapToLoadable(capped, ex, user, 'floor')  // floor so rails can't be breached
```

`snapToLoadable` floors by default and never emits a barbell below 45 lb.

## Applying the migrations

With the Supabase CLI and a linked project:

```bash
supabase db push          # applies the schema then the seed
```

`0001` requires the `auth.users` table (present in any Supabase project). `0002`
is idempotent — safe to re-run after regenerating from the JSON.
